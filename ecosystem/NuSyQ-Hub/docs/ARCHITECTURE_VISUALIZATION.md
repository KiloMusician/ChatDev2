# NuSyQ-Hub Architecture Visualization (Grounded)

Ground truth sources:
- scripts/start_nusyq.py (entrypoint, action routing, path discovery)
- src/spine/spine_manager.py (spine health and snapshot logic)
- state/reports/current_state.md (latest snapshot)
- state/reports/lifecycle_catalog_latest.json (service inventory)
- scripts/start_system.ps1 (system health check)
- start-all-servers.ps1 (frontend server launcher)

## 1) Tripartite Topology (Paths + Roles)

```
NuSyQ-Hub (Spine)      NuSyQ Root (Core)        SimulatedVerse (World)
------------------     -------------------      -----------------------
Path: state/reports/    Path: state/reports/     Path: state/reports/
current_state.md        current_state.md         current_state.md

Orchestration +         Core services            Dev server + simulation
automation +            (MCP, orchestrator,      layer (vite/tsx)
diagnostics + health    notebooks)
```

Path discovery is centralized in `scripts/start_nusyq.py` (load_paths + env/config/cache
resolution). Service roles are enumerated in the lifecycle catalog writer
(`scripts/start_nusyq.py::_handle_lifecycle_catalog`).

## 2) Spine Entry Point and Health Signals

Primary entrypoint: `scripts/start_nusyq.py`
- Initializes the spine via `src/spine/spine_manager.py::initialize_spine`.
- Writes `state/reports/spine_health_snapshot.json` via
  `src/spine/spine_manager.py::export_spine_health`.

Spine health sources (read-only):
- `state/reports/current_state.md` (snapshot summary and repo paths).
- `state/reports/lifecycle_catalog_latest.json` (service catalog snapshot).

The spine status is GREEN only when both files are present and non-empty
(`src/spine/spine_manager.py::_determine_status`).

## 3) Lifecycle Catalog: Service Inventory (Observed State)

The lifecycle catalog is generated by `scripts/start_nusyq.py::lifecycle_catalog`.
It builds a process snapshot and matches running processes against service
definitions (match terms in `_handle_lifecycle_catalog`).

From `state/reports/lifecycle_catalog_latest.json`:
- Active services: 0
- Critical missing services: orchestrator, pu_queue, quest_log_sync, trace_service,
  guild_board_renderer

Service definitions are grounded in `_handle_lifecycle_catalog` and include:
- NuSyQ MCP Server (match terms: mcp_server, nusyq mcp, uvicorn)
- NuSyQ Orchestrator (match terms: start_orchestrator.py, MultiAIOrchestrator)
- SimulatedVerse Dev Server (match terms: server/index.ts, server/minimal-agent-server.ts)
- PU Queue Processor (match terms: unified_pu_queue, pu_queue, processing_units)
- Trace Service (match terms: otel, opentelemetry, trace_service)

## 4) Work Intake and Action Routing

The action router lives in `scripts/start_nusyq.py`:
- KNOWN_ACTIONS enumerates exposed commands.
- dispatch_map wires actions to handlers in `scripts/nusyq_actions/*`.

Core action modules (grounded):
- AI tasks: `scripts/nusyq_actions/ai_actions.py`
- Queue + PU processing: `scripts/nusyq_actions/auto_cycle_steps.py`
- Trace controls: `scripts/nusyq_actions/trace_actions.py`
- Diagnostics: `scripts/nusyq_actions/selfcheck.py`, `scripts/nusyq_actions/doctor.py`

## 5) Orchestration Core

Canonical orchestrator: `src/orchestration/unified_ai_orchestrator.py`
Compatibility shim: `src/orchestration/multi_ai_orchestrator.py`

Activation (module-level only, not OS process startup):
- `src/orchestration/ecosystem_activator.py` discovers and instantiates
  bridges/systems inside the Python process.

## 6) Bootstrap-Adjacent Entry Points

Health check (agent-invokable, read-only): `scripts/start_system.ps1`
Frontend servers: `start-all-servers.ps1`

These do not modify core configs unless explicitly instructed and are safe
to use as visibility checks before starting external services.
