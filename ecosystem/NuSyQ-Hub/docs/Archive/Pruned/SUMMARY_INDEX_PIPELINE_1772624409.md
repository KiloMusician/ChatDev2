# Summary / Report / Analysis Ingestion Pipeline

## Purpose

Activate utilization of existing documentation artifacts (summary, report, analysis, session logs) by indexing them and injecting structured metadata into orchestration and context systems.

## Components

1. `src/tools/summary_indexer.py`
   - Scans repository for patterns: `*SUMMARY*`, `*REPORT*`, `*ANALYSIS*`, `SESSION_*_Summary*`.
   - Extracts metadata: path, category, size, modified timestamp, first heading, Omni/Mega tag traces.
   - Writes JSON index to `docs/Auto/SUMMARY_INDEX.json`.

2. `MultiAIOrchestrator` integration
   - Auto-loads (or generates if absent) the summary index at startup.
   - Injects concise stats into `context_bridge['summary_index']` for task-level context enrichment.

3. `UnifiedDocumentationEngine` enhancement
   - Adds `summary_docs` & `total_summary_docs` per repository in unified documentation map.

## Usage

Generate / refresh index manually:

```bash
python -m src.tools.summary_indexer
```

Run orchestrator (auto-loads index):

```bash
python src/orchestration/multi_ai_orchestrator.py
```

## Retrieval-Augmented Orchestration (RAG)

- `src/tools/summary_retrieval.py` implements a lightweight TF-IDF retrieval engine over `SUMMARY_INDEX.json`.
- On task submission, orchestrator attempts a lightweight retrieval (top-N default 3) and attaches retrieved docs to `task.context['retrieved_summary_docs']` to inform systems (e.g., Ollama) with relevant historical docs.
- This process is feature-flagged and fails-safe: missing index or failures result in no enrichment but don't block tasks.

## Pruning & Archival Lifecycle

- `src/tools/summary_pruner.py` implements heuristics for recommending archival of stale, oversized, or duplicate summary/report/analysis docs.
- Heuristics configurable: `age_days`, `size_threshold_bytes`, `min_duplicate_group`.
- Generates a dry-run JSON plan `docs/Auto/SUMMARY_PRUNE_PLAN.json` by default with recommended archivals and reasons.
- Archival is performed separately through a safe `archive_pruned_files(plan_path, archive_dir)` call which moves files to `docs/Archive/Pruned` (or provided directory), preserving metadata and avoiding deletions.
- Orchestrator integration: `MultiAIOrchestrator.generate_prune_plan(...)` and `archive_pruned_documents(archive_dir)` expose plan generation and archival to higher-level workflows.

### Scheduled Prune Plan Generation

- A scheduled runner is available at `scripts/generate_prune_plan.py`. It reads env variables:
   - `PRUNE_AGE_DAYS` (default 365)
   - `PRUNE_SIZE_BYTES` (default 100000)
   - `PRUNE_MIN_DUP_GROUP` (default 2)
- A GitHub Actions workflow is provided at `.github/workflows/generate-prune-plan.yml` that runs daily (04:00 UTC) and on push to `master`, invoking the runner. It intentionally only generates a plan and does not archive files.
- The script and workflow are intentionally safe: they do not change or commit code; they only write the dry-run plan into `docs/Auto/SUMMARY_PRUNE_PLAN.json`.

#### Configuring schedule and notifications

- To change the schedule, edit the cron expression in `.github/workflows/generate-prune-plan.yml`.
- To adjust pruning thresholds for the job, set repository secrets `PRUNE_AGE_DAYS`, `PRUNE_SIZE_BYTES`, `PRUNE_MIN_DUP_GROUP` in the repository's settings, or update via environment in the workflow.


## Safety & Governance

- Automatic deletion is intentionally disabled; only dry-run plans are generated by default. Archival requires an explicit action.
- All archival actions preserve copies in `docs/Archive/Pruned` and log changes; maintainers should review recommended plans before archiving.

## Maintenance

- Re-run indexer after adding or removing documentation artifacts.
- Non-fatal: failures to load index are logged; orchestration proceeds regardless.
