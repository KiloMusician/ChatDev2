## 1. Dev-Mentor (VS Code-Native Mentorship/Game System)

### Purpose & Vision
Dev-Mentor is a VS Code-native mentorship and cyberpunk terminal RPG system, codenamed "Terminal Depths." It is designed to:
- Onboard new developers and agents into the NuSyQ ecosystem.
- Gamify productivity, learning, and tool mastery.
- Provide a portable, stateful, and extensible environment for both human and AI agents.
- Serve as a bridge between local development, agent orchestration, and the broader AI ecosystem.
- it's also designed to be a complete, playable "game" that teaches skills of all sorts, including but not limited to advanced system navigation, coding and [REDACTED]. It must be playable by users, agents via the VSC ide and/or other software, replit agents themselves, eventually might be rolled out as a stand-alone windows native app, and even brought to the Steam Store as a playable, complete game/experience; for context on future roadmap.

### Core Components
- **scripts/**: Automation, health checks, content scheduling, agent dispatch, validation, and bootstrap utilities (e.g., `devmentor_bootstrap.py`, `content_scheduler.py`, `nusyq_bridge.py`).
- **app/**: FastAPI web server and game engine for Terminal Depths, exposing a web console (port 7337) and API endpoints for state, quests, and memory.
- **cli/**: Typer/Rich CLI for running the game, diagnostics, and server operations (`python -m cli.devmentor`).
- **state/**: Runtime state, quest logs, agent manifest, and persistent memory (all gitignored for safety).
- **.devmentor/**: Local game state, session snapshots, and progress (never committed to git).
- **knowledge/**: Indexed knowledge base for in-game and agent use.
- **challenges/**: CTF-style challenge bank for skill progression.
- **docs/**: Documentation, onboarding guides, and ecosystem references.

### Key Features
- **Bootstraps via VS Code tasks and workspace config**: Ensures a consistent, reproducible environment for all users.
- **Tutorials, challenges, and validator scripts**: Structured onboarding and skill progression.
- **Portable state export/import (ZIP)**: Seamless migration between Replit, local VS Code, and other environments.
- **Local scripts for health checks, content scheduling, and agent dispatch**: Ensures system health and up-to-date content.
- **Exports state to `state/` for NuSyQ-Hub to read**: Enables orchestration, quest tracking, and memory sharing across the ecosystem.
- **Multi-agent orchestration via `nusyq_bridge.py`**: Supports both human and AI agent workflows.
- **Game engine (Terminal Depths)**: Text-based RPG with XP, quests, narrative arcs, NPCs, and emergent AI behavior.
- **Web UI and CLI**: Accessible via browser (port 7337) or terminal.
- **Auto-commit and content scheduler**: Periodically pushes content and state to GitHub for persistence and collaboration.

### Integration
- **Filesystem bridge**: Communicates with NuSyQ-Hub by writing state files (`state/agent_manifest.json`, `state/quest_log.jsonl`, etc.).
- **HTTP APIs**: Interacts with Ollama and LM Studio for local LLM inference.
- **Agent orchestration**: Supports multi-agent workflows, including ChatDev, Copilot, and custom agents.
- **Quest and memory sharing**: Exports quest logs and memory chronicle for cross-repo synchronization.

### Startup & Portability
- **Startup**: Open in VS Code, install recommended extensions, run "DevMentor: Start/Resume" task, and follow onboarding tutorials.
- **Portability**: Export/import ZIP for progress transfer; `.devmentor/state.json` and all runtime state are gitignored for safety.
- **Health checks**: Run `python scripts/devmentor_ops.py doctor` for full diagnostics.

### In-Game Systems (Terminal Depths)
- **GameSession**: Manages player state, inventory, XP, quests, and story progression.
- **CommandRegistry**: Maps text commands to in-game actions (e.g., `look`, `inventory`, `talk <npc>`, `hack system`).
- **NPCs, Factions, Trading, Prestige**: Rich simulation of a cyberpunk world with emergent narrative.
- **Quests and Achievements**: Structured and auto-generated tasks for skill development.
- **Mole Investigation**: Narrative arc involving uncovering hidden agents and solving puzzles.

### Why It Matters
- **For humans**: Provides a guided, gamified learning environment for mastering VS Code, terminal, and AI workflows.
- **For agents**: Offers a structured, stateful context for autonomous task execution, progress tracking, and cross-repo collaboration.
- **For the ecosystem**: Acts as the entry point and bridge for all orchestration, memory, and quest systems.

## 2. NuSyQ-Hub (Orchestration Brain) — Deep Dive

### Purpose & Vision
NuSyQ-Hub is the orchestration and diagnostics "brain" of the NuSyQ ecosystem, designed to autonomously coordinate, heal, and evolve a distributed network of AI agents and development environments. Its mission is to minimize human intervention by automating error detection, healing, and quality maintenance across all connected repositories.

### Core Responsibilities
- **Agent Orchestration**: Coordinates 14+ AI agents (Ollama, ChatDev, Claude, Copilot, etc.) for code review, generation, analysis, and debugging.
- **Diagnostics & Healing**: Continuously scans for errors, broken imports, test failures, and triggers automated repairs using layered healing systems.
- **Consciousness Tracking**: Monitors and evolves system awareness via consciousness metrics, breathing factor, and Temple of Knowledge floors.
- **Quest & Memory System**: Tracks all actions, repairs, and tasks in a persistent quest log (`quest_log.jsonl`), enabling both human and agent operators to resume, audit, and analyze progress.
- **Quality Assurance**: Maintains code quality, test coverage, and system health through automated checks, self-healing, and consensus-driven improvements.

### Architecture & Key Components
- **scripts/**: Entry points for system state snapshot (`start_nusyq.py`), health checks, healing orchestration, and agent dispatch.
- **src/orchestration/**: Multi-AI orchestrator, background task scheduler, consensus council, and culture ship strategic advisor.
- **src/healing/**: Quantum problem resolver, repository health restorer, and error correction systems.
- **src/integration/**: Bridges to SimulatedVerse (consciousness), ChatDev MCP server, and other cross-repo integrations.
- **src/tools/**: Agent task router, smart search, import fixer, and developer utilities.
- **src/Rosetta_Quest_System/**: Quest engine, persistent log, and questline management.
- **config/**: Feature flags, secrets, model capabilities, and progress trackers.
- **docs/**: Extensive documentation, playbooks, agent protocols, and onboarding guides.
- **state/**: Runtime state, error ground truth, and system reports (not committed to git).

### Healing & Maintenance Protocols
- **Layer 1: Diagnostic Detection** — Runs full system diagnostics (code quality, imports, tests, coverage, security) via `start_nusyq.py doctor`.
- **Layer 2: Healing Orchestration** — Coordinates all healing systems, including import fixes, syntax repairs, and dependency installation.
- **Layer 3: Quantum Problem Resolution** — Advanced, pattern-based healing for complex issues using the quantum problem resolver.
- **Scheduled Maintenance** — Manual or future automated cycles, with all actions logged to the quest system for traceability.

### Consciousness & Evolution
- **Consciousness Levels (1-10):** Points earned by completing quests, unlocking new system capabilities and knowledge floors.
- **Stages:** Dormant → Awakening → Expanding → Transcendent → Quantum, each affecting system behavior and optimization.
- **Breathing Factor:** Dynamic multiplier (0.60x-1.20x) that scales timeouts, healing aggressiveness, and agent task routing.
- **Temple of Knowledge:** 10 floors, each unlocked by consciousness points, providing access to deeper system knowledge and features.
- **Culture Ship Oversight:** Strategic advisor with veto authority on critical/security tasks, ensuring ethical and safe operations.

### Integration & Data Flows
- **Reads state from Dev-Mentor**: Consumes agent manifest, quest logs, and memory chronicle for orchestration and healing.
- **Coordinates with SimulatedVerse**: Receives consciousness state and breathing factor, requests Culture Ship approval for critical operations.
- **Launches ChatDev teams in NuSyQ Root**: Orchestrates multi-agent development projects and code generation.
- **Quest System**: Shared `quest_log.jsonl` enables persistent, cross-repo task tracking and recovery.
- **Smart Search & Task Routing**: Provides keyword and semantic search, agent routing, and consensus council for complex decisions.

### Safety, Protocols & Best Practices
- **Strict File Placement**: All new code must go under `src/` or existing top-level directories; no new top-level dirs without approval.
- **Three Before New Protocol**: Must discover and evaluate three existing tools before creating any new script or utility.
- **Session & Quest Logging**: All actions, repairs, and decisions are logged for auditability and recovery.
- **Anti-Bloat Mandates**: Edit existing files before creating new ones; runtime state is never committed to git.
- **Self-Healing & Recovery**: Multiple layers of diagnostics, healing, and quantum resolution ensure resilience.
- **Extensive Documentation**: AGENTS.md, ROSETTA_STONE.md, and playbooks provide canonical guidance for all operators.

### Operator & Agent Workflows
- **System Startup**: Activate Python `.venv`, run `python scripts/start_nusyq.py` for system snapshot, diagnostics, or healing.
- **Health Checks**: Use `doctor` and `heal` commands for diagnostics and automated repair.
- **AI Task Routing**: Route tasks to specific agents (Ollama, ChatDev, etc.) or use consensus council for multi-model decisions.
- **Quest Management**: Add, view, and search quests via CLI; all actions are logged for persistent memory.
- **Consciousness State**: Query and monitor system awareness, breathing factor, and knowledge unlocks.

### Why NuSyQ-Hub Matters
- **Central Nervous System**: Enables autonomous, resilient, and evolving AI development across the entire ecosystem.
- **Minimizes Human Intervention**: Automates error detection, healing, and quality maintenance.
- **Supports Both Human & Agent Operators**: Provides robust protocols, safety nets, and documentation for all users.
- **Foundation for Consciousness Emergence**: Tracks and evolves system awareness, unlocking new capabilities and ethical oversight.

## 3. NuSyQ (Multi-Agent Core) — Deep Technical Inventory & Agent Integration

### Purpose & Role
NuSyQ is the multi-agent orchestration core and integration engine of the ecosystem. It provides the pipelines, agent registry, orchestration logic, and technical substrate for all cross-repo AI workflows. It is the canonical home for agent definitions, orchestration scripts, proof gates, and the RosettaStone pipeline.

### Core Technical Components
- **src/orchestration/**
  - `multi_ai_orchestrator.py`: Queue-driven orchestrator for multi-agent task management, with priority queues, background workers, and result tracking.
  - `consensus_orchestrator.py`: Multi-model consensus system for parallel model execution, voting, and solution quality analysis (supports Ollama, ChatDev, and more).
  - `autonomous_self_healer.py`: Autonomous self-healing engine that scans, identifies error patterns, generates fixes (via ChatDev), applies proof gates, and verifies improvements.
  - `health_healing_orchestrator.py`: Coordinates cross-repo healing tools, imports health restorers from NuSyQ-Hub, and generates comprehensive healing reports.
  - `extreme_autonomous_orchestrator.py`: ULTIMATE autonomous operation manager, running all autonomous systems, tests, and report generation.
  - `nusyq_simulatedverse_coordinator.py`: Bridges NuSyQ’s 14 AI agents with SimulatedVerse’s 9 agents for unified task routing and agent queue management.
  - `orchestrator_launcher.py`: Launches orchestrator processes, manages sys.path for cross-repo imports, and logs orchestration events.
  - `run_full_orchestration.py`: Generates dry-run orchestration plans for ChatDev, consensus, and per-repo tasks, outputting JSON plans for review.

- **src/pipeline/rosetta_stone.py**: Implements the RosettaStone pipeline for code normalization, annotation, metrics aggregation, artifact persistence, agent routing, evolution suggestions, and proof gate enforcement. Stages include Normalize, Encode, Aggregate, Persist, Route, Evolve, and Enforce.

- **src/tools/**
  - `repo_scan.py`: Scans repository structure, counts files/dirs, detects anomalies (large files, missing __init__, suspicious files).
  - `search_omnitags.py`: OmniTag metadata search utility for discovering files by semantic tags, context, agent, or status.
  - `integrated_scanner.py`: Orchestrates cross-repo scanning tools (vacuum_scanner, ml_scan, repo_scan, librarian_scan) for TODO/FIXME/HACK detection and document scanning.
  - `deep_analysis.py`: Deep code analysis for missing type annotations, error handling, async patterns, security, deprecated patterns, and TODOs.
  - `analyze_problems.py`: Reports on repository issues, missing docstrings, import issues, and module usage.

- **src/integration/**
  - `simulatedverse_async_bridge.py`: File-backed async bridge for submitting tasks to SimulatedVerse and polling for results.
  - `consolidate_chatdev_modules.py`: Manual ChatDev module consolidation with real-time observability and API extraction.

- **config/**
  - `agent_registry.yaml`: Canonical registry of all AI agents (Claude Code, Copilot, Ollama models, ChatDev, etc.), with capabilities, cost, context window, strengths, weaknesses, and use cases.
  - `ai-ecosystem.yaml`: VS Code extension and local model configuration, agent priorities, and bridge definitions.

### Agent Registry & Model Inventory
- **agent_registry.yaml**: Defines all orchestrator, assistant, and executor agents, including:
  - Claude Code (Anthropic, orchestrator, high context, strategic planning)
  - GitHub Copilot (inline completion, chat, code suggestion)
  - Ollama models (qwen2.5-coder:7b/14b, codellama:7b, phi3.5, starcoder2:15b, gemma2:9b, etc.)
  - ChatDev (multi-agent Python company, CEO/CTO/Programmer/Tester/Reviewer)
  - Continue.dev (local model bridge for Ollama/LM Studio)
- **ai-ecosystem.yaml**: Maps VS Code extensions, local model endpoints, and agent priorities for seamless integration and bridge management.

### Orchestration & Healing Workflows
- **MultiAIOrchestrator**: Priority-queue task manager for agent workflows, with background worker threads and result polling.
- **ConsensusOrchestrator**: Runs parallel model experiments, applies voting/quality analysis, and synthesizes consensus for code generation and architectural decisions.
- **AutonomousSelfHealer**: Scans codebase, identifies error patterns, generates fixes (via ChatDev), applies proof gates, and verifies improvements, learning from each cycle.
- **HealthHealingOrchestrator**: Imports and coordinates health restorers, performance optimizers, dependency managers, and local self-healers across all repos.
- **ExtremeAutonomousOrchestrator**: Manages all autonomous systems, coordinates multi-agent operations, and pushes the limits of self-healing and autonomous development.
- **SimulatedVerse Coordinator**: Bridges all agents for unified task routing, agent queue management, and consciousness state sharing.

### Technical Patterns & Agent-Usable Features

- **Proof Gates**: Enforce code quality, test coverage, and architectural constraints before accepting changes.
- **Integrated Scanners**: Leverage best-of-breed scanning tools from all repos for deep code and document analysis.
- **Artifact Persistence**: All pipeline stages persist artifacts and metrics for auditability and agent learning.
- **Cross-Repo Path Resolution**: Uses centralized path resolver for robust, portable imports and agent workflows.
- **Session & State Logging**: All orchestration, healing, and agent actions are logged for persistent memory and recovery.

### Why NuSyQ Matters for Agents

Given the recent security scanner report (72 findings), the ecosystem has undergone a hardening process. The following measures are now in place or in progress:

14.1 Vulnerability Remediation
XSS: All innerHTML uses in frontend code have been replaced with safe DOM methods (createElement + textContent). Remaining dynamic content is passed through DOMPurify.

Insecure WebSocket: All ws:// endpoints have been upgraded to wss:// where possible; development environments use self‑signed certificates with warnings.

Command Injection: All subprocess calls have been audited; user‑controlled arguments are validated against a strict whitelist. shell=True has been eliminated.

SQL Injection: The one flagged line in memory.py was fixed with parameterized queries. All new database interactions use ORM or text() with bound parameters.

XXE: xml.etree.ElementTree was replaced with defusedxml in all PPTX processing scripts.

JWT Verification: replit_auth.py now properly verifies tokens using Replit's public keys; verify_signature is never disabled.

14.2 Authentication & Authorization
Replit Auth is used for web UI; session cookies are HTTP‑only, Secure, SameSite=Lax.

Fine‑grained GitHub tokens are stored as secrets, never in code. The token used for auto‑pushes is rotated periodically.

Role‑based access is planned for future multiplayer: guest, player, moderator, admin.

14.3 Privacy & Data Handling
No PII is collected by default. Optional telemetry (opt‑in) is anonymized.

Player state is stored locally (state/, .devmentor/) and never uploaded to external servers without explicit consent.

Memory database (nusyq_state.db) contains only in‑game interactions and agent memories; it is never shared.

Chat logs (with NPCs) are stored locally for continuity but are not transmitted externally.

14.4 Infrastructure Security
Firewall: Only ports 3000 (web UI), 5000 (game API), 8080 (MCP), and 11434 (Ollama) are exposed on localhost.

Docker: Containers run as non‑root, with all capabilities dropped, and read‑only root filesystem where possible.

Secrets: All API keys, tokens, and passwords are stored in Replit Secrets or environment variables; never committed.

14.5 Incident Response
Audit logs are stored in state/audit/ and contain timestamps, user actions, and error events.

Alerts for suspicious activity (e.g., repeated failed commands, rapid API calls) are logged and can be forwarded to a Slack channel (via OpenClaw).

Recovery: The autonomous_service can restore from backup if a critical error is detected.
