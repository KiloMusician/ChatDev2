# Rosetta Stone

Artifact authority: hand-authored canonical contract

## Purpose

`NuSyQ` owns the distributed Rosetta bundle for the local ecosystem. Rosetta is not a monolithic repo or a single generated file. It is the canonical semantic map that lets humans, agents, extensions, and scripts boot into the same reality without rediscovery.

## Control-Plane Read Precedence

All ecosystem consumers should load context in this order:

1. `state/boot/rosetta_bootstrap.json`
2. `state/registry.json`
3. `state/reports/control_plane_snapshot.json`
4. focused feed artifacts such as quest logs, repair queues, and error reports
5. docs only as fallback or deep expansion

## Repo Roles

- `CONCEPT`
  - kernel / machine-governance layer
  - pressure, mode, maintenance, diagnostics, execution guardrails
- `Dev-Mentor / TerminalDepths`
  - developer guidance and task plane
  - workflow coaching, MCP/game surfaces, human-agent command surfaces
- `SimulatedVerse`
  - simulation and runtime layer
  - ChatDev-adjacent UX, patch-bay behavior, Culture Ship runtime owner
- `NuSyQ-Hub`
  - orchestration and control-plane layer
  - GitNexus, Nogic, doctor/healing, strategic coordination, Culture Ship control owner
- `NuSyQ`
  - Rosetta and proof layer
  - distributed memory, bundle generation, canonical boot capsule

## Culture Ship Contract

Culture Ship has dual authority and must be represented that way everywhere:

- `runtime_owner: simulatedverse`
- `control_owner: nusyq_hub`

Implications:

- SimulatedVerse exports runtime truth via `state/culture_ship_runtime_descriptor.json`
- NuSyQ-Hub consumes that runtime truth into the control-plane snapshot and presents the strategic view
- no active surface should imply that Culture Ship has only one owner

  - generated human attach twin

## Anti-Rediscovery Rules

- If a port, bridge, command, repo role, or extension ownership changes, update the bundle inputs immediately.
- If a capability is legacy but still present, mark it in the deprecation registry instead of leaving it ambiguous.
- If a working surface already exposes structured truth, consume that before adding new prose.
- Heavy docs should be referenced from the bootstrap, not copied into it.

## Canonical State Surface

- `state/reports/control_plane_snapshot.json`
  - should expose both live service health and the current storage/state map
  - use it to find repo-owned ledgers, cache DBs, runtime memory stores, and legacy archive pressure before broad repo inspection

## Generated Bundle Outputs

Generated by `scripts/build_rosetta_bootstrap.py`:

- `state/registry.json`
- `state/deprecation_registry.json`
- `state/boot/rosetta_bootstrap.json`
- `state/boot/ROSETTA_BOOT.txt`

The current control-plane snapshot is refreshed by `NuSyQ-Hub`, and the current Culture Ship runtime descriptor is refreshed by `SimulatedVerse`.
